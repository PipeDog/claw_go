# 01｜项目现状与产品定位

## 一、项目基本判断

`Claw Go` 当前是一个 **macOS 优先的桌面 AI 助手**，目标不是让普通用户直接理解 OpenClaw 的底层概念，而是通过更友好的桌面端界面，把复杂的 CLI、Gateway、Session、Profile 等能力包装成可操作的任务型体验。

从 README 可以看到，项目当前产品方向已经比较明确：

- 主信息架构：**首页 / 我的任务 / 帮助中心 / 设置**
- 默认隐藏技术细节
- 高级工具集中放在设置页里
- 首页强调“大输入框 + 常用动作卡”
- 任务页强调“任务摘要 + 可展开过程”

这说明当前产品方向并不是做一个开发者专属工具，而是希望做一个 **对普通用户更友好的 OpenClaw 桌面工作台**。

---

## 二、当前项目的已有优点

### 1. 产品方向是清楚的

从仓库文档和页面命名看，项目并不是无目标堆功能，而是已经有一个较清晰的产品判断：

- 普通用户优先
- 技术细节默认折叠
- 日常任务入口优先于系统配置入口
- 结果呈现优先于底层运行过程

这是一个很重要的优势，因为很多桌面 AI 工具的问题不是做不出来，而是一开始就把自己做成“技术配置面板”。

### 2. Flutter 项目结构已经相对规范

当前 `lib/` 目录基本遵循了：

- `app`
- `business`
- `component`
- `foundation`

四层分层结构；同时业务侧也基本按照 `page / view / view_model / repository / model` 的 MVVM 方式组织。

这意味着：

- 当前项目不是散乱堆页面
- 架构已经具备继续扩展的基础
- 后续做页面拆分、模块扩展、功能沉淀的成本相对可控

### 3. 已经在做“技术能力 → 用户体验”的包装

从首页、任务页、设置页的实现看，项目已经在努力把底层 OpenClaw 能力转成更自然的产品体验：

- 聊天页强调任务提交和聊天连续性
- 任务页强调任务列表与详情联动
- 设置页把高级工具放到二级展开
- Runtime / Gateway / Session 等能力已经有专门的承载位置

这说明这个项目已经不是“纯壳子”，而是在认真做桌面产品化。

---

## 三、当前项目的结构现状

## 1. 信息架构现状

当前 RootPage 中实际挂载了这些页面：

- Chat（首页）
- Overview（概览）
- Sessions（任务 / 会话）
- Logs（日志）
- Setup（环境准备）
- Profiles（环境 / Profile）
- Console（控制台）
- Config（设置）
- Docs（帮助）
- Markdown Test（Markdown 测试页）

从实现上看，当前项目其实存在两层结构：

### 面向用户的一层
- Chat
- Sessions
- Settings
- Help

### 面向内部 / 高级用户的一层
- Overview
- Logs
- Setup
- Profiles
- Console
- Markdown Test

这说明产品方向和代码实现之间，已经开始出现一点点“目标是轻量，但实现还偏工作台化”的张力。

---

## 2. 首页现状

首页当前已经具备几个关键能力：

- 任务输入
- Agent / Model 选择
- 聊天消息流展示
- 最近会话列表
- 任务回退 / 恢复链路

这说明首页不只是一个输入框，而是一个初步成形的任务工作区。

但同时也意味着：首页的复杂度已经明显上升，离“普通用户一眼能懂”的目标开始变远。

---

## 3. 设置页现状

设置页当前已经同时承担：

- 偏好设置
- Runtime 信息展示
- 高级工具入口
- Setup / Workspace / Console 三个技术页面的承载容器

这让设置页很强大，但也容易变重。

一旦后续能力继续增加，设置页很容易变成“什么都往里放”的总容器。

---

## 4. Runtime 层现状

`default_openclaw_runtime_adapter.dart` 已经承担了大量职责，包括：

- 环境探测
- Node 版本探测
- CLI 调用
- Gateway 状态检查
- Gateway 启停与重启
- Gateway 连接
- Gateway 聊天发送与事件监听
- Config 路径识别
- 鉴权读取
- 命令执行与进程管理

这说明能力做得比较全，但也说明 Runtime Adapter 已经变成一个非常重的核心类。

---

## 四、当前问题的本质判断

当前项目最大的问题，不是没有方向，也不是架构混乱，而是：

> **产品想做“普通用户友好”，但当前实现还保留了较强的“开发工作台”气质。**

再直白一点说：

- 方向是对的
- 基础也不错
- 但要真正变成一个“好用的桌面产品”，还需要进一步做减法、收口和重新梳理层次

---

## 五、当前阶段最值得确认的产品问题

后续继续推进前，建议团队先明确以下问题：

1. 当前版本主要面向谁？
   - OpenClaw 已有用户？
   - 完全不懂 CLI 的普通用户？
   - 内部团队演示与验证用户？

2. 首页的目标到底是什么？
   - 是“最快发起一个任务”？
   - 是“持续和 AI 对话”？
   - 还是“管理多个任务与会话”？

3. 设置页是否应该继续承载这么多技术页面？
   - 还是应该把“设置”和“高级工具”做更明确的层级拆分？

4. 当前版本更应该优先优化什么？
   - 新用户首次可用性？
   - 日常任务效率？
   - 底层运行稳定性？
   - 技术架构可维护性？

这些问题不先统一，后续很容易在“继续加能力”和“产品变复杂”之间来回摇摆。
