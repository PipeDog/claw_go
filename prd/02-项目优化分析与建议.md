# 02｜项目优化分析与建议

## 一、结论先说

`Claw Go` 当前不是“做得乱”，而是已经走到了一个典型节点：

- 底层能力越来越完整
- 页面越来越多
- 用户体验目标也越来越明确

这个阶段最需要做的，不是继续横向加页面，而是把项目从“可运行的桌面工作台”进一步推向“真正可理解、可持续迭代的桌面产品”。

---

## 二、产品层优化建议

## 1. 首页需要进一步做减法

### 当前观察
首页当前同时承载：

- 聊天消息流
- Runtime 选择
- 最近会话
- 任务回退
- 任务提交

能力已经不少。

### 问题
对普通用户来说，首页最重要的问题应该是：

> “我现在要做什么？”

如果首页同时承载过多运行时概念，会提高理解门槛。

### 建议
- 把首页进一步明确为“任务发起 + 当前任务继续处理”的核心场景。
- 将 Agent / Model 等能力弱化成“高级可见”或“二级展开”。
- 对最近会话区做更强的信息筛选，只保留用户最关心的内容，如：
  - 最近正在进行的任务
  - 最近完成的任务
  - 最近失败的任务

### 预期收益
- 首页更像产品入口，而不是控制台。
- 降低首次使用门槛。
- 更贴近 README 中“普通用户优先”的目标。

---

## 2. 信息架构需要继续收口

### 当前观察
RootPage 中当前挂载的页面已经较多，包括：

- chat
- overview
- sessions
- logs
- setup
- profiles
- console
- config
- docs
- markdown_test

### 问题
从产品视角看，这些页面的角色并不处于同一层级。

例如：
- `markdown_test` 更像开发调试页
- `logs` 和 `console` 更像高级工具页
- `setup` 和 `profiles` 更像环境管理页

如果这些页面长期并列出现在主导航结构中，会破坏主产品叙事。

### 建议
建议把导航重构为两层：

#### 主导航
- 首页
- 我的任务
- 帮助中心
- 设置

#### 高级工具区
- 环境准备
- Profile 管理
- 控制台
- 日志
- 开发调试页

### 预期收益
- 主结构更稳定
- 用户理解成本更低
- 后续继续扩展时不容易失控

---

## 3. 设置页建议拆分“设置”和“高级工具”心智

### 当前观察
SettingsPage 当前既是偏好设置页，又是高级工具入口容器。

### 问题
这会让“设置”承担过多职责：

- 保存主题、语言、Shell、Node 路径
- 展示 Runtime 信息
- 控制是否显示高级工具
- 承载 Setup / Workspace / Console 三个复杂页面

### 建议
长期建议拆分为：

- **设置**：只放用户偏好与当前运行环境摘要
- **高级工具**：单独作为一个技术工作区入口

如果当前不想大改，也建议至少在文案与分区上更明确：

- 基础设置
- 环境状态
- 高级工具

### 预期收益
- 设置页更像“偏好中心”
- 高级能力不再挤压主设置体验

---

## 三、交互层优化建议

## 1. 任务状态表达还可以更清楚

### 当前观察
任务页已有 running / failed / completed 等状态，但从产品体验角度还可以进一步强化。

### 建议
- 对运行中任务增加更清晰的动态反馈，如阶段标签或简短状态文案。
- 对失败任务增加更明显的“原因摘要 + 下一步建议”。
- 对已完成任务增加更清晰的结果摘要，而不只是过程记录。

### 预期收益
用户更容易理解“现在卡在哪”“接下来怎么办”。

---

## 2. 首页与任务页之间的关系建议再梳理

### 当前观察
现在首页和任务页都在承载任务相关能力。

### 风险
如果边界不够清晰，会导致：

- 首页越来越重
- 任务页越来越像重复页面
- 用户不知道该在哪处理什么

### 建议
建议明确分工：

- 首页：发起任务 + 查看当前活跃任务
- 任务页：看全部任务 + 管理历史 + 查看详细过程

### 预期收益
两个页面职责更清楚，不容易互相侵蚀。

---

## 四、架构层优化建议

## 1. `DefaultOpenClawRuntimeAdapter` 过重，建议拆分

### 当前观察
这个类已经承担了：

- CLI 探测
- Node 探测
- Config 探测
- Gateway 启停
- Gateway 连接
- 聊天会话管理
- 进程执行
- 鉴权配置读取
- 输出解析

### 问题
单类职责过重，会带来：

- 理解成本高
- 维护风险高
- 后续改动影响面过大
- 难以独立验证各个子能力

### 建议拆分方向
建议至少拆成以下几个能力模块：

1. `runtime_detection_service`
   - CLI / Node / Config 探测

2. `gateway_runtime_service`
   - Gateway 状态、启动、重启、连接

3. `gateway_chat_service`
   - chat.send / chat.abort / event listen

4. `runtime_process_service`
   - 命令执行、ManagedProcess 生命周期管理

5. `runtime_auth_service`
   - config / env / file 中的鉴权读取

### 预期收益
- 结构更清晰
- 代码 review 更容易
- 未来替换底层实现时更安全

---

## 2. ViewModel 初始化模式需要更稳定

### 当前观察
多个 ViewModel 在构造函数中直接 `unawaited(load())`。

### 风险
这种写法虽然方便，但会带来几个问题：

- 初始化顺序不好控制
- 多个页面联动时更难判断刷新时机
- 重建 Provider 时容易出现重复加载
- 调试状态问题时难追踪

### 建议
- 对关键 ViewModel 建立更明确的初始化策略。
- 对依赖上游状态的 ViewModel，避免在构造时直接做副作用式加载。
- 可以考虑把“初始化”和“刷新”动作显式化。

### 预期收益
- 状态流更稳定
- Provider 生命周期更清晰

---

## 3. RootPage 的页面注册方式可以进一步抽象

### 当前观察
当前 `RootPage` 中直接维护了长列表 `items` + `orderedIds`。

### 问题
随着页面继续增加，会带来：

- 页面配置分散
- `items` 与 `orderedIds` 需要人工保持同步
- 导航变更时容易漏改

### 建议
- 将导航配置提取为统一配置源。
- `id / group / label / icon / pageBuilder` 统一管理。
- 避免在一个页面类里同时维护展示配置和跳转索引逻辑。

### 预期收益
- 导航结构更可维护
- 页面增删更安全

---

## 五、数据与状态层优化建议

## 1. 任务数据模型可以继续区分“过程数据”和“结果数据”

### 当前观察
`AssistantTask` 当前同时承载：

- transcript
- chatMessages
- summary
- failureMessage
- status

### 问题
随着后续需求增加，任务模型可能会越来越重。

### 建议
后续可考虑更明确地分层：

- 任务基础信息
- 聊天消息流
- 运行过程日志
- 任务结果摘要
- 用户可见状态

### 预期收益
- 页面渲染更容易按需取数
- 任务详情和聊天区职责更清晰

---

## 2. 持久化策略建议逐步升级

### 当前观察
当前已有本地存储基础，但还看不出更细粒度的缓存治理策略。

### 建议
后续可逐步明确：

- 哪些是设置类持久化
- 哪些是任务历史持久化
- 哪些是短期运行态，不应长期落盘
- 哪些是敏感信息，需要更严格隔离

### 预期收益
- 降低状态污染
- 降低历史数据膨胀风险
- 提升安全性

---

## 六、可维护性优化建议

## 1. 建议补一份“页面级产品说明”

当前仓库已有 README、AGENTS、架构规范，但缺少页面级产品文档。

建议后续逐步补：

- 首页目标是什么
- 任务页目标是什么
- 设置页目标是什么
- 帮助页目标是什么
- 高级工具页面为什么存在

这样后续做改动时，不会只围绕代码实现，而是能回到页面目标。

---

## 2. 建议补一份“导航与信息架构说明”

目前从代码可以看出导航结构，但没有专门的产品文档解释：

- 为什么这样分组
- 哪些页面面对普通用户
- 哪些页面面对高级用户
- 哪些页面属于开发调试

建议把这个内容单独沉淀成文档，避免后续页面越加越乱。

---

## 七、优先级建议

## P0（优先做）

1. 建立 PRD 文档目录并持续沉淀
2. 收口主导航结构，明确主导航与高级工具的边界
3. 明确首页、任务页、设置页三者的职责边界
4. 给 Runtime Adapter 制定拆分类方案

## P1（第二阶段）

1. 拆分 Runtime Adapter
2. 优化任务状态表达与失败反馈
3. 减少首页对底层概念的直接暴露
4. 让设置页更像设置页，而不是综合技术容器

## P2（后续优化）

1. 升级任务持久化与缓存策略
2. 细化页面级 PRD
3. 补齐高级工具区与普通用户区的体验分层
4. 沉淀统一的导航配置系统

---

## 八、最后的总体判断

这个项目当前最值得肯定的点，是：

- 有明确方向
- 有相对规范的技术架构
- 已经在做产品化包装

最需要优化的点，不是“有没有能力”，而是：

> **要进一步把复杂能力收口成更清楚的产品层级，让普通用户入口更轻，让高级能力边界更稳，让核心 Runtime 代码结构更可维护。**

如果这几个点先做好，`Claw Go` 后续不管是继续扩页面、加能力，还是做真正可交付的桌面产品，都会更稳。
