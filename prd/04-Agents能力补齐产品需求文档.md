# 04｜Agents 能力补齐产品需求文档

## 一、文档背景

本文用于指导 `Claw Go` 后续逐步补齐 `/agents` 相关能力，并统一产品目标、信息架构与阶段交付边界。

本次参考来源主要包括：

1. 当前仓库已有产品结构与页面能力。
2. 本地页面 `http://127.0.0.1:18789/agents` 的实际可见内容。
3. `/agents` 当前处于 **未授权态**，可确认的前置链路包括：
   - `WebSocket URL`
   - `Gateway Token`
   - `Password (not stored)`
   - `Connect`
   - 提示：`unauthorized: gateway token missing ...`
   - 连接说明：
     - `openclaw gateway run`
     - `openclaw dashboard --no-open`
     - 粘贴 URL 与 Token，或直接打开带 token 的 URL

> 结论：`/agents` 的所有正式能力都建立在“Gateway 运行 + WebSocket 可达 + Token 鉴权成功”之上，因此 **连接与鉴权** 必须是第一阶段能力，而不是附属能力。

---

## 二、产品目标

### 1. 产品目标

将当前偏“环境配置 + 会话控制”的桌面工作台，逐步升级为一个可管理、多 Agent 协作、具备清晰可视化反馈的 **Agent 控制中心**。

### 2. 用户目标

用户来到 `Agents` 相关区域时，核心诉求不是看技术名词，而是完成以下几件事：

1. 确认当前是否已连接到 Gateway。
2. 明确有哪些 Agent 可以使用。
3. 快速理解每个 Agent 当前状态、能力和上下文。
4. 发起、恢复、观察、回退某个 Agent 会话。
5. 在出现问题时，快速定位是连接问题、鉴权问题还是执行问题。

### 3. 业务目标

1. 将 OpenClaw 的 Agent 能力产品化，而不是停留在命令行映射。
2. 建立一套稳定的 Agent 工作流：连接 → 选择 Agent → 发起任务 → 查看过程 → 管理会话。
3. 逐步减少“用户必须理解底层 CLI 才能使用”的门槛。

---

## 三、问题本质

当前仓库已经具备：

- Chat：对话与任务发起
- Sessions：任务 / 会话列表
- Logs：诊断日志
- Profiles：Profile 管理
- Console：运行控制

但这些能力还没有被组织成一个完整的 `Agents` 产品心智，问题主要有三点：

### 1. 连接前置能力没有单独成体系

从 `/agents` 可见页面可以确认，用户真正进入 Agent 能力之前，必须先完成：

- Gateway 启动
- WebSocket 地址确认
- Token 获取与填充
- 连接状态确认

如果这个链路不稳定，后续所有 Agent 能力都会变成“假入口”。

### 2. Agent 资产管理还不够完整

当前项目已支持 Agent / Model 选择，但还缺少一个完整的 Agent 资产视图，例如：

- Agent 列表
- 默认 Agent
- Agent 描述与用途
- Agent 绑定 workspace
- Agent 当前可用性 / 在线状态

### 3. Agent 执行过程没有形成闭环

当前聊天、任务、日志能力已经存在，但仍偏分散：

- Chat 更像日常操作入口
- Sessions 更像任务工作台
- Logs 更像诊断面板

后续需要把这三者真正围绕 Agent 串起来。

---

## 四、产品定位

`Agents` 不是一个单独的“设置页”，也不是一个纯技术控制台。

它更准确的定位应当是：

> **围绕 Agent 连接、选择、运行、观察、恢复与诊断的一体化工作区。**

因此后续能力建设必须围绕以下五层展开：

1. 连接层：Gateway / WebSocket / Token / Password
2. 资产层：Agent 列表、默认项、workspace、模型、描述
3. 执行层：发起任务、发送消息、调用工具、运行状态
4. 会话层：最近会话、历史会话、回退、恢复、分支
5. 诊断层：日志、错误、连接异常、授权异常

---

## 五、目标用户与场景

### 1. 目标用户

1. 已经在使用 OpenClaw 的桌面用户
2. 需要管理多个 Agent / Profile / Workspace 的进阶用户
3. 希望减少 CLI 依赖、转向桌面 GUI 的用户

### 2. 核心场景

#### 场景 A：首次连接 Gateway
用户第一次进入 Agent 工作区，需要被清晰引导完成连接。

#### 场景 B：查看可用 Agent 并选择当前工作 Agent
用户需要快速知道“我现在可以用谁”。

#### 场景 C：基于某个 Agent 发起任务
用户从 Chat 或 Agent 工作区直接发起任务，并看到过程与结果。

#### 场景 D：恢复或回退到某次会话
用户希望延续上下文，而不是每次重新开始。

#### 场景 E：排查连接与执行问题
用户需要通过日志与状态提示，快速知道问题卡在哪一层。

## 六、信息架构建议

后续建议将 `Agents` 相关能力拆成以下结构：

### 1. Agent Overview（总览）
用于回答三个问题：

- 是否已连接
- 当前默认 Agent 是谁
- 现在有哪些可操作能力

建议信息块：
- 连接状态卡
- 默认 Agent 卡
- 最近运行记录
- 最近异常 / 告警

### 2. Agent Directory（Agent 列表）
用于集中管理 Agent 资产。

建议信息块：
- Agent 基础信息
- workspace / model / profile 关联
- 默认标记
- 可用性状态
- 快捷动作（设为当前、发起聊天、查看会话）

### 3. Agent Workspace（Agent 工作区）
用于承接实际操作。

建议信息块：
- 当前 Agent 上下文摘要
- 输入区
- 最近会话
- 会话详情
- 可回退节点

### 4. Agent Diagnostics（Agent 诊断）
用于承接连接、鉴权、执行过程异常。

建议信息块：
- 连接状态
- 最近错误
- 平铺日志流
- 复制 / 清空 / 过滤能力

## 七、功能需求拆解

## 7.1 连接与鉴权

### 目标
把 `/agents` 未授权页暴露出的前置链路产品化。

### 功能点
1. WebSocket URL 输入与展示
2. Gateway Token 输入与展示状态
3. Password 临时输入（不落盘）
4. Connect / Disconnect
5. 当前连接状态提示
6. 未授权错误提示与修复指引
7. 一键复制连接命令 / 指引

### 关键交互要求
1. 用户必须一眼知道当前是“未连接 / 连接中 / 已连接 / 鉴权失败”。
2. 鉴权失败时不能只展示错误字符串，要同时给出修复动作。
3. 输入项要支持记忆合理内容，但 Password 不落本地持久化。

---

## 7.2 Agent 资产管理

### 目标
把当前已有的 Agent / Model 选择能力升级为完整的 Agent 管理能力。

### 功能点
1. 展示 Agent 列表
2. 展示默认 Agent
3. 展示 Agent 绑定 workspace
4. 展示 Agent 绑定模型 / provider
5. 支持设为当前 Agent
6. 支持查看 Agent 描述与用途
7. 支持展示 Agent 可用状态

### 关键交互要求
1. 默认 Agent 必须有明确标识。
2. 当前正在使用的 Agent 必须有明确状态。
3. Agent 卡片的信息密度要高，但层次要清晰，避免再做成配置表单感。

---

## 7.3 Agent 会话管理

### 目标
让 Agent 的聊天、任务、最近会话、历史回退形成一个统一体验。

### 功能点
1. 当前 Agent 的聊天列表
2. 最近会话列表
3. 查看某次会话详情
4. 回退到某次会话
5. 从某次会话继续分支
6. 查看该 Agent 的历史上下文摘要

### 关键交互要求
1. 当前 Agent 聊天记录与最近会话必须明确区分。
2. 会话详情区必须承接足够多的信息密度，避免纵向堆叠导致有效内容太少。
3. 回退操作必须让用户清楚自己回到了哪一个上下文节点。

---

## 7.4 Agent 执行与反馈

### 目标
让用户能看到 Agent 任务的执行过程，而不是只看到最终结果。

### 功能点
1. 任务运行中状态
2. 执行成功 / 失败状态
3. 结果摘要
4. 失败原因摘要
5. 当前任务与会话的关联关系
6. 后续扩展的工具调用 / 事件流展示区

### 关键交互要求
1. 运行中、已完成、失败必须一眼可区分。
2. 不同结果态要有明确下一步动作，例如“继续追问”“查看日志”“重新连接”。

---

## 7.5 诊断与恢复

### 目标
降低用户排障成本。

### 功能点
1. 平铺日志流
2. 关键类别高亮（异常 / 请求 / 网关 / 成功）
3. 清空日志
4. 复制日志
5. 后续扩展过滤 / 搜索
6. 连接异常与执行异常分类展示

### 关键交互要求
1. 日志不是纯文本堆积，而应支持快速扫描。
2. 日志页既要高信息密度，也要保留可复制、可阅读能力。

---

## 八、与当前项目的映射关系

### 现有可复用能力
1. `Chat`：可作为 Agent Workspace 的主入口
2. `Sessions`：可作为 Agent 会话管理基础
3. `Logs`：可作为 Agent Diagnostics 基础
4. `Profiles`：可作为 Agent 运行环境与 workspace 配置基础
5. `Console`：可作为 Gateway 与运行控制基础

### 当前缺口
1. 缺统一的 Agent 连接页 / 连接流
2. 缺统一的 Agent 列表与资产视图
3. 缺以 Agent 为中心的信息组织方式
4. 缺更明确的“连接失败 → 修复动作”链路
5. 缺 authenticated `/agents` 页面能力对照清单

---

## 九、阶段建议

### P0：先补连接与结构
- 补连接与鉴权工作流
- 统一 Agent 相关入口与文案
- 把当前日志、会话、聊天整理成 Agent 语义

### P1：补 Agent 资产与会话工作区
- 做 Agent 列表
- 做 Agent 状态卡
- 做 Agent 工作区聚合页

### P2：补高级执行能力
- 工具调用事件
- 更细粒度的任务状态
- 更强的诊断与过滤能力

---

## 十、风险与依赖

### 1. 当前最大风险
`/agents` 页面当前为未授权态，因此我们现在拿到的只是“前置连接层信息”，还没有拿到 authenticated 态的完整功能明细。

### 2. 依赖项
1. 本地 Gateway 运行稳定
2. Token 获取链路稳定
3. WebSocket 通信协议清晰
4. OpenClaw authenticated Dashboard 能力可观测

### 3. 后续补充要求
拿到 authenticated `/agents` 页面后，需要追加一轮文档补充，重点校准：

1. 实际 Agent 列表字段
2. Agent 状态字段
3. 会话 / 运行 / 工具事件结构
4. 页面真实导航与操作流

---

## 十一、最终判断

当前最合理的推进方式不是马上“把 /agents 全量照搬”，而是：

> **先把连接与鉴权做稳，再把 Agent 资产和会话闭环做完整，最后再补高级执行与诊断能力。**

这样既符合 `/agents` 页面当前暴露出的真实前置条件，也更符合 `Claw Go` 逐步产品化的最短路径。
