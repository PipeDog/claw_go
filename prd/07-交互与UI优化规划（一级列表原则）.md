# 07｜交互与 UI 优化规划（一级列表原则）

## 一、核心原则

本文只围绕一个约束展开：

> **不要出现多级列表。**

如果一个页面的信息过多，不要继续向下套“列表里的列表”“卡片里的列表”“侧栏里的列表”。

更优的办法只有两种：

1. **收敛成一级列表**
2. **在当前页面内部增加 Tab / Segment / Scope 切换**

这条原则的底层原因很简单：
- 多级列表让用户难以建立空间记忆
- 多层折叠会导致内容分散、滚动距离变长
- 用户无法快速判断“当前所见列表属于哪一层”

---

## 二、当前交互问题总览

## 2.1 一级导航层级过多

当前左侧侧边栏同时承载：
- 主任务入口
- 技术工作区入口
- 状态与诊断入口
- 开发测试入口

### 问题
这不是“信息丰富”，而是“层级未收口”。

### 优化方向
先收敛一级导航，再在页面内部用二级 Tab 承接细分能力。

---

## 2.2 Settings 页存在“嵌套页面容器”问题

当前 Settings 页内部有：
- 偏好设置
- 运行时摘要
- 高级工具开关
- 高级工具内部 Tab（Setup / Profiles / Console）

### 问题
这相当于：
- 一级 Tab：Config
- 二级 Tab：Setup / Profiles / Console
- 二级 Tab 内再进入各自复杂页面

这会形成明显的多层导航嵌套。

### 优化方向
Settings 页应收敛为：
- 偏好
- 当前环境摘要

不要再把 Setup / Profiles / Console 嵌进去。

---

## 2.3 Chat 页的内容密度与角色还可继续收敛

当前 Chat 页包含：
- Agent / Model 工具条
- 聊天主区
- 输入区
- 最近会话

### 问题
Chat 是高频页，但右侧最近会话仍然在和主聊天区竞争注意力。

### 优化方向
Chat 页继续遵守“单主任务原则”：
- 主体永远是“当前 Agent 的聊天与输入”
- 最近会话只能做一级简版列表
- 如果用户想看更多，直接跳 Sessions

### 建议
- 保留 3~5 条一级列表
- 不要再在 Chat 页做“会话卡片展开二级信息”
- 如果需要更多过滤，放到 Sessions 页内部切换

---

## 2.4 Sessions 页已经改善，但还可继续收口

当前 Sessions 已经具备：
- 当前 Agent / 全部任务切换
- 左右布局
- 左侧列表、右侧详情

### 当前优点
这个方向是对的，因为它已经从“纵向堆叠”转向“单一工作台”。

### 还可优化点
右侧详情区仍然可以进一步结构化，避免一口气平铺所有内容。

### 建议
在详情区内部增加顶部二级 Tab：
- 摘要
- 过程
- 结果

而不是把 prompt、summary、failure、process 全部继续下堆。

### 原则说明
这里不是多级列表，而是：
- 左侧一级列表：任务列表
- 右侧内容区：Tab 切换不同信息视图

这是允许且推荐的。

---

## 2.5 Agents 页不应继续增长为“超级页”

当前 Agents 页已有：
- 概览
- Agent 列表
- 最近会话预览

### 风险
后续如果继续往下堆：连接、目录、会话、日志、能力说明，就会再次形成大而全页面。

### 建议
Agents 页内部尽快升级为页面内 Tab：
- Directory
- Workspace
- Diagnostics

### 具体建议
#### Directory
- Agent 资产列表
- 默认 / 当前标识
- workspace / model / 描述

#### Workspace
- 当前 Agent 会话与工作区摘要

#### Diagnostics
- 当前 Agent 相关的连接与问题摘要

这样可以保证：
- 左侧一级导航只保留 `Agents`
- 页内通过 Tab 切换具体子能力
- 避免 Agents 页面继续无限纵向增长

---

## 2.6 Environments 应合并 Setup 与 Profiles

### 当前问题
Setup 与 Profiles 分成两个一级页，但用户心智里它们都属于“环境”。

### 建议
未来合并为一个一级 Tab：`Environments`

页内使用 Tab：
- Discover（发现 / 导入）
- Manage（管理 / 编辑 / 校验）

### 好处
- 一级导航减负
- 用户不会再纠结“导入环境去 Setup，修改环境去 Profiles”
- 同时满足一级列表原则，不需要再额外嵌套页面

---

## 2.7 Logs 不应独立承担一个完整一级导航

### 当前问题
Logs 是系统诊断类能力，但它的使用频率和认知优先级都低于 Chat / Sessions / Agents。

### 建议
Logs 逐步降级为：
- Connect 页中的 `Diagnostics` 子 Tab
或
- Agents 页中的 `Diagnostics` 子 Tab（如果明确是 Agent 诊断）

### 原因
日志不是用户目标，而是用户在遇到问题时才会进入的辅助信息层。

---

## 三、推荐的页面结构

## 3.1 推荐一级导航

1. Chat
2. Sessions
3. Agents
4. Connect
5. Environments
6. Settings
7. Docs

---

## 3.2 推荐页面内 Tab 结构

### Chat
无需额外 Tab，保持单主任务页。

### Sessions
页内二级 Tab：
- 摘要
- 过程
- 结果

### Agents
页内二级 Tab：
- Directory
- Workspace
- Diagnostics

### Connect
页内二级 Tab：
- Connection
- Status
- Diagnostics

### Environments
页内二级 Tab：
- Discover
- Manage

### Settings
页内二级 Tab 可选，但必须轻量：
- Preferences
- Runtime Summary

不要再嵌入 Setup / Profiles / Console 的完整页面。

---

## 四、当前项目的具体优化建议

## P0：先做信息架构减法
1. 评估移除一级导航中的 Markdown Test
2. 评估移除一级导航中的 Overview
3. 评估将 Logs 并入 Connect / Diagnostics
4. 评估将 Setup + Profiles 合并为 Environments

## P1：做页面内 Tab 收口
1. Agents 页加页内 Tab
2. Sessions 详情区加页内 Tab
3. Connect 页加页内 Tab

## P2：做视觉统一
1. 统一页面头部结构
2. 统一一级列表卡片样式
3. 统一详情区结构
4. 统一状态标签、指标卡、摘要卡的尺寸与层级

---

## 五、反模式清单

后续设计和开发中，应尽量避免以下反模式：

1. 一级导航里再放“开发调试页”
2. 同一能力既有一级入口，又嵌在另一个 Tab 里再次出现
3. 一个页面先列表，再列表项里嵌另一个列表
4. 一个页面同时出现多个同级主标题块，表达同一主题
5. 用纵向无限堆叠替代页面内 Tab 切换

---

## 六、最终判断

从第一性原理看，用户真正需要的不是“更多入口”，而是：

1. 能快速判断应该去哪个一级页
2. 进入页面后，能快速判断当前处于哪一个工作层
3. 看到的内容是一级列表，而不是层层嵌套后的信息迷宫

因此当前最优策略不是继续平铺内容，而是：

> **减少一级入口，减少多级列表，把复杂性收进页面内 Tab。**

这条原则应作为后续 UI / 交互优化的硬约束。
