# 05｜Agents 能力分阶段路线图

## 一、路线目标

本路线图用于指导 `Claw Go` 在后续几个迭代中，逐步补齐 `Agents` 相关能力，并确保每一阶段都能形成清晰、可交付、可验证的产品增量。

---

## 二、推进原则

1. **先打通前置链路，再扩能力面**
   - 先做 Gateway / Token / WebSocket / 鉴权
   - 再做 Agent 资产与工作区

2. **先形成闭环，再做高级能力**
   - 优先保证连接、发起、查看、恢复、诊断可闭环
   - 不提前扩张到复杂编排能力

3. **以当前项目可复用能力为基础演进**
   - 不推翻 Chat / Sessions / Logs / Profiles / Console
   - 而是围绕它们做 Agent 化整合

---

## 三、阶段拆分

## P0｜建立 Agent 前置能力与文档基线

### 阶段目标
让用户能明确完成“连接到 Agent 能力”这件事。

### 重点范围
1. Gateway 状态可视化
2. WebSocket URL / Token / Password 连接表单
3. Connect / Disconnect
4. 连接错误提示与修复指引
5. PRD 文档与信息架构同步

### 交付项
1. Agent 连接入口
2. 连接状态卡片
3. 未授权 / 鉴权失败提示
4. 连接成功后的状态反馈
5. 文档基线（本轮已建立）

### 验收标准
1. 用户无需看命令行即可理解如何连接。
2. 连接失败时能明确知道失败原因与下一步动作。
3. 产品内所有 Agent 文案开始统一。

---

## P1｜补齐 Agent 资产管理

### 阶段目标
让用户看清“有哪些 Agent，可用哪个 Agent，当前正在用哪个 Agent”。

### 重点范围
1. Agent 列表
2. 默认 Agent 标记
3. Agent 绑定 workspace / model / profile 信息
4. 当前 Agent 状态提示
5. Agent 选择与切换体验优化

### 交付项
1. Agent 列表页或 Agent 资产区
2. Agent 卡片组件
3. 当前 Agent 概览区
4. 当前 Agent 快捷动作入口

### 验收标准
1. 用户可以快速识别默认 Agent 与当前 Agent。
2. 用户可以在不打开配置文件的情况下理解 Agent 差异。
3. Agent 选择操作能够稳定反映到 Chat / Sessions。

---

## P2｜补齐 Agent 工作区与会话闭环

### 阶段目标
让 Agent 的聊天、会话、回退、分支形成一个顺滑工作流。

### 重点范围
1. 当前 Agent 聊天记录
2. 最近会话与历史会话
3. 会话详情区
4. 回退到某次会话
5. 从某次会话继续分支

### 交付项
1. Agent 工作区布局
2. 聊天与会话联动机制
3. 清晰的回退状态提示
4. 更高信息密度的会话详情呈现

### 验收标准
1. 用户能分清聊天记录与最近会话。
2. 用户能理解自己回退到了哪一次上下文。
3. 用户能连续使用 Agent，而不是每次重头开始。

---

## P3｜补齐执行反馈与诊断闭环

### 阶段目标
让 Agent 从“能用”提升到“可观察、可排障”。

### 重点范围
1. 任务状态展示
2. 结果摘要与失败摘要
3. 日志分类高亮
4. 日志复制 / 清空 / 后续过滤
5. 连接问题与执行问题区分

### 交付项
1. 任务状态与结果反馈组件
2. 诊断日志工作区
3. 异常修复入口
4. 后续过滤 / 搜索预留结构

### 验收标准
1. 用户在失败时可以快速定位问题层级。
2. 日志页可读性明显优于纯文本堆叠。
3. 产品内“运行态”与“诊断态”衔接清晰。

---

## P4｜补 authenticated `/agents` 对齐与高级能力

### 阶段目标
在拿到 authenticated `/agents` 页面之后，校准真实能力清单，并补齐高阶交互。

### 重点范围
1. 校准真实页面字段与模块
2. 校准真实状态机
3. 对齐工具调用、事件流、任务流
4. 对齐更多 Agent 控制能力

### 交付项
1. `/agents` authenticated 态能力映射表
2. 页面差异分析文档
3. 高级能力补齐计划

### 验收标准
1. 当前产品结构与真实 `/agents` 能力形成稳定映射。
2. 不再依赖猜测推进后续功能。
3. 后续版本规划可以按真实能力拆解。

---

## 四、版本优先级建议

### 必做（P0 + P1）
1. 连接与鉴权
2. Agent 列表与当前 Agent 状态
3. Agent 文案与入口统一

### 高优先（P2）
1. 会话闭环
2. 回退 / 分支能力打磨
3. 工作区布局优化

### 中优先（P3）
1. 结果反馈
2. 更强诊断
3. 日志过滤与搜索

### 后置（P4）
1. authenticated `/agents` 全面对齐
2. 高级执行流与更多控制能力

---

## 五、建议执行顺序

### 第 1 步
先把 Agent 连接能力单独抽清楚，形成统一入口。

### 第 2 步
基于当前配置与 Profile 数据，补 Agent 列表与默认状态。

### 第 3 步
继续强化 Chat / Sessions 的 Agent 中心化体验。

### 第 4 步
升级 Logs，使其成为真正的 Agent 诊断面板。

### 第 5 步
在拿到 authenticated `/agents` 页面后，做一次真实能力校准。

---

## 六、依赖与风险

### 依赖
1. OpenClaw Gateway 可稳定运行
2. Token 化 Dashboard 链路可用
3. 本地协议和字段可继续观测

### 风险
1. 当前 `/agents` 页面未授权，真实能力面仍不完整
2. 如果先做高级能力，容易与真实 Dashboard 偏离
3. 如果不先收敛入口，后续页面会继续分散

---

## 七、最终建议

后续推进 `Agents` 能力时，推荐遵循这条主线：

> **连接先行，资产补齐，会话成环，诊断收口，最后再对齐高级能力。**

这样可以确保每一阶段都在给用户增加真实可用价值，而不是单纯叠页面或堆技术入口。
